void dailyDiskCleanup() {

== Node Types

=== Trigger Nodes

Start your workflows:

[cols="1,3"]
|===
|Node |Description

|**Schedule**
|Time-based: cron syntax, intervals, specific times

|**File Watch**
|Triggers when files change, appear, or disappear

|**System Event**
|Process starts/stops, user login, network change, hardware event

|**Webhook**
|HTTP endpoint triggers workflow (integration with external tools)

|**Manual**
|Run on-demand via CLI or UI button

|**Condition Watch**
|Continuously monitor condition, trigger when true (e.g., "CPU > 80%")
|===

=== Operation Nodes

Do the work:

[cols="2,3"]
|===
|Category |Operations

|**Network**
|Diagnose, repair, test connectivity, check bandwidth, measure latency, restart interfaces

|**Disk**
|Check space, find large files, clean temp/cache, analyze usage, verify integrity

|**Security**
|Run audits, scan for vulnerabilities, check permissions, verify configs, rotate credentials

|**Performance**
|Monitor CPU/RAM, identify heavy processes, optimize, kill stuck processes, gather metrics

|**Backup**
|Backup files/configs, verify backups, restore, sync to remote, clean old backups

|**Package Management**
|Update packages, check for CVEs, install/remove, verify integrity, rollback updates

|**Logs**
|Search logs, analyze errors, rotate logs, archive, send to aggregator

|**Custom Commands**
|Run arbitrary shell commands (with safety constraints)
|===

=== Logic Nodes

Control flow:

[cols="1,3"]
|===
|Node |Description

|**If/Then/Else**
|Branch on condition (supports comparisons, regex, status checks)

|**Switch**
|Multi-way branch (like switch/case in programming)

|**Loop**
|Iterate over items (files, processes, results), with break conditions

|**Delay**
|Wait for time or condition before continuing

|**Parallel**
|Run multiple branches simultaneously, optionally wait for all to complete

|**Try/Catch**
|Handle errors gracefully (run fallback on failure)

|**Debounce**
|Prevent rapid repeated execution (e.g., file watch triggering too often)

|**Rate Limit**
|Throttle execution (max N times per hour)
|===

=== Output Nodes

Communicate results:

[cols="1,3"]
|===
|Node |Description

|**Send Email**
|Email results, reports, alerts

|**Notification**
|Desktop notification, system tray alert

|**Log**
|Write to log file or syslog

|**Webhook**
|POST results to HTTP endpoint (Slack, Discord, custom API)

|**File Write**
|Save results to file (JSON, CSV, text)

|**Database**
|Store in SQLite, PostgreSQL, etc.

|**Dashboard**
|Send metrics to AmbientOps Ward dashboard

|**Human Approval**
|Pause workflow, ask for approval before continuing
|===

**Visual Flow:**

[source,text]

**What It Does:**

* Triggered manually before risky system changes
* Backs up /etc and other config directories
* Attempts the changes
* Verifies system is still healthy (network up, services running)

* If success: keeps backup for 7 days

**Use Case:** Fearless config editing with safety net.

=== Philosophy Alignment

Composer follows AmbientOps principles:

* **Evidence-first**: Workflows show what they will do before doing it (test mode)
* **Non-destructive**: Automatic backups, reversible actions
* **Explicit approval**: Workflows require user approval for dangerous operations
* **No security holes**: Cannot open firewall ports without explicit consent node

See `docs/AMBIENTOPS_INTEGRATION.adoc` for technical details.

=== Backend

* **Language**: D (robust, cross-platform, good metaprogramming)
* **Execution**: Compiles workflows to D scripts (JIT or ahead-of-time)
* **Storage**: SQLite (workflow definitions, versions, execution logs)
* **Scheduler**: Built-in cron-like scheduler (no external dependencies)

=== Frontend

**Terminal UI (default):**

* Library: notcurses (rich TUI, mouse support, works over SSH)
* Responsive: Works on small terminals (80x24) and large screens
* Keyboard-first: Vim-like navigation (also mouse support)

**Web UI (optional):**

* Lightweight HTTP server (D's vibe.d or similar)
* No external dependencies (bundled assets)
* Works on localhost only by default (security)
* Can enable network access with `--listen 0.0.0.0` (if trusted network)

=== Generated Scripts

Workflows compile to readable D code:

* Uses AmbientOps libraries (same APIs as manual coding)
* Includes comments explaining each step
* Can be edited manually (advanced users)
* Changes sync back to visual editor (best-effort)

Example round-trip:

1. Build visual workflow in Composer
2. Export to D code, edit in text editor
3. Re-import to Composer (shows visual representation)

=== Requirements

* D compiler (dmd, ldc, or gdc)
* notcurses (for TUI) or web browser (for Web UI)
* AmbientOps tools (network-ambulance, traffic-conditioner, etc.)
* SQLite (bundled with most systems)

== Safety Features

=== Execution Sandboxing

Workflows cannot:

* Delete files outside specified paths
* Modify system files without explicit permission nodes
* Open network ports (unless explicit node)
* Install packages (unless explicit node)
* Run as root (unless explicit sudo node + approval)

Dangerous operations require:

1. Explicit node in workflow
2. User approval at design time
3. Confirmation at runtime (for destructive actions)

=== Validation

Before deploying:

* âœ“ Syntax check (no invalid node connections)
* âœ“ Logic check (no infinite loops, unreachable nodes)
* âœ“ Permissions check (has required privileges)
* âœ“ Dependency check (required tools installed)
* âœ“ Safety check (flags dangerous patterns)

Fix issues before deployment.

=== Approval Gates

Workflows can pause for human approval:

[source,text]
----
Backup Configs â†’ Apply Changes â†’ Wait for Approval â†’ Deploy to Production
----

"Wait for Approval" node sends notification, workflow pauses until approved.

=== Execution Limits

Prevent runaway workflows:

* Max execution time (default: 1 hour)
* Max loop iterations (default: 10,000)
* Max concurrent workflows (default: 10)
* Rate limiting (prevent triggering too often)

Configurable per workflow.
